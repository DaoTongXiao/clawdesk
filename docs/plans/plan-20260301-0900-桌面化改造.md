# 变更计划：桌面化改造

## 1. 变更目标与明确边界
- 目标：在不破坏现有网页功能前提下，新增 tauri 桌面壳层，使同一套前端可在网页与桌面双运行。
- 不在本次范围：不重写聊天业务逻辑、不改协议、不改数据模型、不改现有网页接口契约。

## 2. 变更文件清单

- src-tauri/tauri.conf.json | 变更类型：增 | 变更行数预估：+120 | 主要改动描述：窗口、权限、打包、安全白名单
- src-tauri/src/main.rs | 变更类型：增 | 变更行数预估：+80 | 主要改动描述：桌面入口与命令注册
- src-tauri/src/commands/detect_gateway.rs | 变更类型：增 | 变更行数预估：+90 | 主要改动描述：本地网关配置探测命令

- src/lib/store.tsx | 变更类型：改 | 变更行数预估：+35/-10 | 主要改动描述：自动探测逻辑改为桌面优先、网页兜底

## 3. 风险点清单
1. 风险描述：桌面打包与现有服务端路由冲突  
   可能后果：桌面包启动白屏或接口不可用  
   缓解措施：桌面端改走本地命令，网页端保留原路由兜底。
2. 风险描述：本地配置读取涉及敏感令牌  
   可能后果：令牌泄漏  
   缓解措施：仅读取固定路径，最小字段返回，禁止日志打印令牌。
3. 风险描述：权限配置过宽  
   可能后果：攻击面扩大  
   缓解措施：最小权限原则，仅开放必要能力。
4. 风险描述：双端分支逻辑回归  
   可能后果：网页端功能退化  
   缓解措施：保留原逻辑兜底并补关键路径验证。
5. 风险描述：构建链新增带来调试不稳定  
   可能后果：本地开发体验下降  
   缓解措施：保留原网页脚本为默认路径，桌面脚本增量引入。

## 4. 回滚策略
- 方案一：桌面化改造单独提交，使用 git revert 一键回滚。
- 方案二：网页构建与启动脚本保持不变，若桌面异常立即切回网页发布流程。
- 方案三：桌面改动集中在 src-tauri 与桥接文件，按文件清单快速回退。

## 5. 主推荐方案
- 方案：双运行时并存。网页端保持原样，桌面端新增 tauri 壳层与本地命令；前端仅做最小桥接改造。
- 原因：变更体积最小、安全边界清晰、回滚成本最低、风格一致性最好。

## 6. 保守备选方案
- 仅先落地桌面壳初始化与启动链路，不接入本地自动探测；桌面端先手工填写网关地址与令牌。
- 取舍：风险更低、改动更小，但首次体验较弱。
