# 变更计划：引入 sqlite3 持久化会话记录

## 1. 变更目标与明确边界
- 目标：在桌面运行时引入 sqlite3，将设置、会话、消息持久化到本地数据库文件；网页运行时保留现有本地存储兜底。
- 不在本次变更范围：不改网关协议、不改界面交互语义、不改对外接口参数结构、不引入云同步。

## 3. 风险点清单
1. 风险描述：表结构或事务处理不当  
   可能后果：消息丢失或顺序错乱  
   缓解措施：写入统一事务、消息按时间正序读取、删除操作显式事务化。
2. 风险描述：桌面数据库路径不可写  
   可能后果：初始化失败导致无法使用  
   缓解措施：固定到应用数据目录并在失败时返回明确错误，由前端降级到已有流程。
3. 风险描述：前后端字段映射不一致  
   可能后果：界面读写异常  
   缓解措施：命令返回统一驼峰字段，并复用既有类型结构。
4. 风险描述：网页端回归  
   可能后果：非桌面环境数据能力受损  
   缓解措施：保留 Dexie 兜底逻辑，仅在桌面运行时切换 sqlite3。

## 4. 回滚策略
- 方案一：按本次提交单独回滚（git revert）。
- 方案二：前端存储层保留旧实现，紧急情况下强制走旧存储分支。
- 方案三：仅回退 src-tauri 持久化模块并移除命令注册，快速恢复现状。

## 5. 主推荐方案
- 方案：新增桌面 sqlite3 持久化模块与命令，前端以运行时分支调用；网页维持原存储兜底。
- 选择原因：改动边界最清晰、兼容性最好、回滚成本最低、对现有业务破坏最小。

## 6. 保守备选方案
- 方案：只将消息和会话切到 sqlite3，设置仍保留旧存储。
- 取舍：风险更低，但会产生双存储并存的短期维护成本。
